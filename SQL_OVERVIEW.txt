# SQL Overview

This document summarizes the purpose and structure of the SQL script in [test.sql](test.sql).

## Overall Purpose

The script is a KPI reporting workbook for telecom activity. It produces daily and regional summaries for:

- Daily active subscribers
- Recharge
- Loan
- Repayment
- Direct bundle purchase
- Gross additions (GA)
- Voice
- Data
- SMS

The queries are written for a reporting window and mostly aggregate by `ID_DATE` and, where needed, by region.

## Main Tables Used

- `TIGIST_KEBEDE.MW_DAILY_ACTIVE_SUBS` - daily active subscriber base
- `ACTIVATION_MSISDN` - MSISDN to executive region mapping for daily active subscriber regional reporting
- `BI.FCT_FIN_BLNC_MVMN` - financial movements used for recharge, loan, repayment, and bundle purchase reporting
- `BI.REF_SUBSCRIBER` - subscriber reference data such as activation date and staff flag
- `BI.REF_LOCATION` - location and region reference data
- `BI.FCT_SUBSCRIBER_ACTIVATION` - activation event details and creation location
- `BI.FCT_TRAFFIC` - traffic usage for voice, data, and SMS
- `BI.REF_ACCOUNT` - account classification used in paid/free logic
- `BI.REF_PRODUCT` - product classification used in paid/free logic

## Business Classifications

### Service and Traffic Types

The script uses different traffic or movement codes to separate business events:

- `service_type <> 'B.VOICE_INC_ONNET'` to exclude on-net voice increment activity from daily active subscriber counts
- `ID_TRAFFIC_TYPE = 1` for voice
- `ID_TRAFFIC_TYPE = 2` for SMS
- `ID_TRAFFIC_TYPE = 3` for data
- `ID_TRAFFIC_TYPE = 4` for direct bundle purchase
- `ID_TRAFFIC_TYPE = 6` for loan and part of repayment logic
- `ID_TRAFFIC_TYPE = 7` for recharge
- `ID_TRAFFIC_TYPE = 9` for new repayment logic

### Region Mapping

Most regional queries convert raw region names into ordered reporting labels:

- `A.West Addis`
- `B.East Addis`
- `C.Central`
- `D.South`
- `E.North West`
- `F.North East`
- `G.East 1`
- `H.East 2`
- `I.West`
- `J.North`
- `K.Afar`
- `L.Unknown` or `L.Uknown` for unmatched values

This is done with repeated `CASE` expressions on `EXEC_REGION` or `DS_EXEC_REGION`.

### Paid vs Free Logic

The voice, data, and SMS sections contain commented and active classification rules that split traffic into paid and free usage using:

- `RA.DS_ACCOUNT_GROUP`
- `RP.X_BUNDLE_TYPE`
- `RA.DS_ACCOUNT_ID`
- `FT.X_ADDITIONAL_INFO`
- `FT.ID_ROAMING_TYPE`

These rules distinguish account usage, bundles, out-of-bundle traffic, J4U traffic, and roaming traffic.

## Query Families

### 1. Daily Active Subscribers

These queries count distinct MSISDNs in `MW_DAILY_ACTIVE_SUBS` for a single day, a 30-day window, and a 90-day window. The regional version joins to `ACTIVATION_MSISDN` and groups by executive region.

### 2. Recharge

Recharge reporting is built from `BI.FCT_FIN_BLNC_MVMN` and includes:

- Successful transactions only (`ID_RESULTS = 1`)
- Recharge traffic type and related movement filters
- Exclusion of staff CRM transactions in the direct bundle-style logic

The regional version also calculates traffic value using `VL_ACCNT_BLNC / 100 / 1.2575`.

### 3. Loan

Loan reporting focuses on:

- `ID_TRAFFIC_TYPE = 6`
- `ID_FIN_BLNC_MVMN_TYPE = 4`
- Successful records only

It reports both subscriber counts and transaction quantity, with a regional version that also calculates traffic amount.

### 4. Repayment

Repayment reporting mixes two repayment definitions:

- New repayment logic with `ID_TRAFFIC_TYPE = 9`
- Old repayment logic with `ID_TRAFFIC_TYPE = 6` and `ID_FIN_BLNC_MVMN_TYPE = 28`

It also calculates:

- Amount paid
- Principal paid amount
- Unpaid principal amount
- Paid service fee
- Unpaid service fee

### 5. Direct Bundle Purchase

This section uses traffic type 4 and filters bundle-related transactions using:

- Account ids `211` and `212`
- Transaction id patterns such as `%PRETUPS%` and `%CRM%`
- Exclusion of staff CRM transactions

### 6. GA

Gross additions are derived from `BI.REF_SUBSCRIBER` using `DS_ACTIVATION_DATE`. The regional version joins activation and location data to group GA by region.

### 7. Voice, Data, and SMS

The traffic tables report usage by day and region:

- Voice uses `ID_TRAFFIC_TYPE = 1`
- SMS uses `ID_TRAFFIC_TYPE = 2`
- Data uses `ID_TRAFFIC_TYPE = 3`

These queries count subscribers and sum usage, with some sections currently simplified and others preserving more detailed paid/free logic as comments.

## Common Reporting Pattern

Across the file, the SQL follows a repeated pattern:

1. Choose a business event type
2. Filter by date range
3. Keep only successful records when applicable
4. Exclude special cases such as staff CRM traffic or on-net voice increment usage
5. Aggregate by day and optionally by region
6. Convert raw codes into business-friendly labels

## Notes

- The script is heavily duplicated in its region mapping logic, so a reusable reference table or view would reduce repetition.
- Some sections contain commented-out logic, which suggests the business rules are still evolving or were simplified for temporary reporting.
- The date values are hardcoded in the current script and should be parameterized if the queries will be reused regularly.

## Local Mock Setup

To practice the workbook without the company database, use the local DuckDB mock warehouse in [mock_warehouse.py](mock_warehouse.py).

1. Install the dependency from [requirements.txt](requirements.txt).
2. Build the synthetic database with `python mock_warehouse.py build`.
3. Run the workbook with `python mock_warehouse.py run --sql test2.sql --out outputs`.
4. The runner strips Oracle optimizer hints and rewrites the small set of Oracle date helpers used in the workbook.

The mock data covers the tables and columns referenced by the workbook and is seeded so the output is reproducible.
